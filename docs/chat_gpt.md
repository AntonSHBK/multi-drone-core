Конкретный backend должен вызываться не “где попало по коду”, а в **строго определённых точках жизненного цикла команды и состояния**. Тогда `core` остаётся единым, а backend подключается только там, где действительно нужно взаимодействие с внешним миром.

Структурно это выглядит так: в `core` есть некий центральный объект, условно **координатор исполнения**. Он работает с:

* входной командой;
* текущим внутренним состоянием дрона;
* выбранным backend;
* результатом выполнения.

Сам backend выбирается **один раз при инициализации экземпляра system/core**, на основании конфигурации узла. Например:

* для одного борта включён `MavlinkBackend`;
* для другого — `RosBackend`;
* для SITL-теста — `RosBackend` или `MavlinkBackend` в симуляционном окружении.

То есть backend не выбирается заново на каждую команду, если у тебя один активный способ связи с FC. Обычно `core` стартует, читает конфигурацию и получает ссылку на конкретную реализацию backend. После этого все команды исполняются через один и тот же объект backend.

Правильная логика вызова выглядит так.

Сначала приходит сообщение из `transmission`.
`system` передаёт его в `core`.
`core` преобразует внешнее сообщение в внутреннюю команду.
Потом `core` не сам отправляет что-то в UART или ROS topic, а вызывает у backend унифицированную операцию исполнения.

Например, на концептуальном уровне так:

`core.execute(command) -> backend.execute(command)`

Но перед этим `core` делает свою часть работы:

* парсит команду;
* валидирует параметры;
* проверяет, допустима ли команда в текущем состоянии;
* переводит внутренний FSM в промежуточное состояние, если нужно.

А уже потом передаёт управление backend.

Это первый момент времени, когда вызывается backend: **момент фактического исполнения принятой команды**.

Теперь по шагам на примерах.

---

### Пример 1. Команда takeoff

На НСУ сформирована команда `takeoff`.
Она в JSON проходит через `transmission`, приходит на дрон и попадает в `core`.

Дальше по шагам:

**1. Декодирование**
`core` получает JSON и преобразует его во внутреннюю команду `TakeoffCommand(height=10)`.

**2. Проверка состояния**
`core` смотрит на свой внутренний `DroneState`:

* есть ли связь;
* доступен ли FC;
* не в failsafe ли система;
* не выполняется ли уже другая несовместимая команда.

Если всё допустимо, `core` переводит внутренний автомат, например:
`idle -> arming_pending -> takeoff_pending`

**3. Вызов backend**
Вот здесь `core` вызывает backend:

* если активен `MavlinkBackend`, он отправляет нужную MAVLink-команду;
* если активен `RosBackend`, он публикует нужное сообщение в ROS topic или вызывает action/service.

То есть в этот момент времени вызов выглядит логически так:

* `backend.arm()`
* `backend.takeoff(target_altitude=10)`

или

* `backend.execute(TakeoffCommand(...))`

Это зависит от выбранного стиля интерфейса, но смысл один: **именно backend переводит доменную команду в конкретный способ исполнения**.

**4. Асинхронное ожидание результата**
После вызова backend `core` не считает команду завершённой мгновенно.
Он ждёт обновления состояния.

И вот тут возникает второй важный момент вызова backend.

---

### Когда backend вызывается второй раз

Backend нужен не только для отправки команды, но и для **обновления состояния дрона**.

Это происходит постоянно, пока работает система.

Но механизм разный:

* у `RosBackend` состояние обновляется через callbacks подписок;
* у `MavlinkBackend` состояние обновляется через чтение потока MAVLink-сообщений.

Однако наверху это должно выглядеть одинаково: backend поставляет в `core` обновлённый `DroneState`.

То есть в runtime backend работает в двух ролях:

**1. Исполнитель команд**
Когда `core` явно вызывает его на выполнение команды.

**2. Поставщик состояния**
Когда backend непрерывно обновляет внутреннее состояние системы.

Для команды `takeoff` это выглядит так:

* `core` вызвал `backend.takeoff(...)`;
* backend начал отправку в FC;
* дальше backend получает признаки выполнения:

  * armed state;
  * текущую высоту;
  * режим полёта;
  * ошибки / ack / failsafe;
* по этим данным `core` понимает:

  * команда принята;
  * взлёт начался;
  * высота достигнута;
  * можно перевести состояние в `loiter` или `hold`.

То есть команда завершается не в момент вызова backend, а в момент, когда **через backend пришло подтверждённое изменение состояния**.

---

### Пример 2. Команда go_to_point

Приходит команда:
`go_to(x, y, z, yaw)`

Жизненный цикл такой же.

**1. core**
Разбирает JSON в `GoToPointCommand`.

**2. core**
Проверяет, что текущее состояние позволяет движение:

* drone armed;
* режим допускает внешнее управление;
* нет active failsafe.

**3. core**
При необходимости меняет свой FSM:

* `hold -> offboard_preparing`
* или `mission -> trajectory_update`

**4. core вызывает backend**
Вот тут различие реализаций особенно видно.

Для `MavlinkBackend`:

* отправка setpoint через MAVLink;
* возможно, предварительное переключение режима в guided/offboard-эквивалент;
* возможно, циклическая передача setpoint.

Для `RosBackend`:

* публикация target pose / trajectory / offboard control mode в нужные ROS-топики;
* возможно, параллельная публикация heartbeat / keepalive для offboard.

То есть backend вызывается в тот момент, когда уже нужно перевести абстрактную команду перемещения в конкретные действия стека PX4/ArduPilot/ROS.

**5. core наблюдает за прогрессом**
Через обновляемое backend состояние:

* текущая позиция;
* активный режим;
* target reached;
* отклонение;
* таймаут.

Когда условие достижения выполнено, `core` завершает команду.

---

### Где именно backend “живёт” по времени

Если смотреть по времени работы процесса, то backend участвует в двух режимах.

**Первый режим — фоновый, постоянный**
Сразу после старта `core` инициализирует backend.
Backend:

* подключается к ROS или UART;
* запускает подписки / циклы чтения;
* начинает поставлять состояние наверх.

То есть backend начинает работать **ещё до первой команды**.

**Второй режим — реактивный, по событию**
Когда приходит новая команда, `core` вызывает у backend операцию исполнения.

Иными словами:

* backend **запускается один раз** при инициализации;
* backend **обновляет состояние постоянно**;
* backend **исполняет действия точечно**, когда `core` делегирует конкретную команду.

---

### Почему backend не должен вызываться напрямую из system

Потому что тогда `system` начнёт знать:

* какой именно backend активен;
* какие у него методы;
* как устроен ROS или MAVLink.

Это разрушит идею унификации.

Правильная цепочка такая:

`system -> core -> backend`

А не:

`system -> RosBackend`
или
`system -> MavlinkBackend`

`system` вообще не должен знать, кто под капотом исполняет команду. Для него есть только `core`.

---

### Упрощённая структурная схема

Можно мыслить это так.

**На этапе старта:**

1. `system` создаёт `core`
2. `system` по конфигурации подставляет нужный backend в `core`
3. `core` инициализирует backend
4. backend устанавливает соединение и начинает обновлять состояние

**На этапе команды:**

1. `transmission` доставляет сообщение
2. `system` передаёт его в `core`
3. `core` создаёт внутреннюю команду
4. `core` валидирует команду
5. `core` вызывает backend для исполнения
6. backend отправляет низкоуровневое действие
7. backend продолжает обновлять состояние
8. `core` по состоянию завершает или отменяет команду

---

### Пример различия по вызовам backend

Для одной и той же команды `takeoff`:

**Через MAVLink**

* `core` решил выполнить `takeoff`
* вызывает `MavlinkBackend`
* backend:

  * проверяет наличие соединения с FC
  * отправляет arm
  * отправляет takeoff command
  * читает heartbeat и position
  * сообщает обновлённое состояние в `core`

**Через ROS**

* `core` решил выполнить `takeoff`
* вызывает `RosBackend`
* backend:

  * публикует команды в ROS
  * подписывается на state/pose/status
  * обновляет единый `DroneState`
  * сообщает обновлённое состояние в `core`

Снаружи для `core` это одна и та же история:
он не знает деталей, он знает только, что команда отправлена и потом по состоянию отслеживается её выполнение.
